# Bug Bounty Automation System

## Архитектура

```
┌─────────────────────────────────────────────────────┐
│                  Claude (Main Session)               │
│  - Координатор всех агентов                         │
│  - Финальное решение о сабмите                      │
│  - Верификация находок                              │
└──────────────┬──────────────────────────────────────┘
               │ запускает
    ┌──────────┼──────────┐
    ▼          ▼          ▼
┌───────┐ ┌───────┐ ┌──────────┐
│Explore│ │General│ │  Plan    │
│Агент  │ │Агент  │ │ Агент    │
│(код)  │ │(web)  │ │(отчёты)  │
└───────┘ └───────┘ └──────────┘
```

## Агенты и их роли

| Агент | Тип | Инструменты | Роль |
|-------|-----|-------------|------|
| Code Scanner | Explore | Read, Grep, Glob, Bash | Анализ кода цели |
| Research | general-purpose | WebSearch, WebFetch + все | Проверка CVE, huntr |
| Verifier | general-purpose | Read, Bash, Grep | Верификация PoC |
| Report Writer | general-purpose | Write | Оформление репорта |

## Процесс поиска уязвимости

```
1. SCAN (параллельно)
   ├─ semgrep --config p/security-audit target/
   └─ agent(Explore): читает подозрительные файлы

2. VERIFY (параллельно, 2 независимых агента)
   ├─ agent_1: "Подтверди что эта строка кода уязвима"  
   └─ agent_2: "Найди существующие CVE для этого паттерна"
   
3. REPORT (только если оба агента согласны)
   └─ agent(Plan): генерирует huntr-репорт по шаблону

4. SUBMIT (только человек!)
   └─ Родион: вручную копирует на huntr.com
```

## Мониторинг

- huntr-monitor.sh — каждые 30 мин (claudeclaw cron)
- Уведомления в Telegram при новых таргетах
- Логи: ~/projects/bounty-lab/tools/logs/

## Контроль токенов

Проблема: агенты потребляют много токенов.
Решение:
- Explore агент вместо General для поиска в коде (в 3x дешевле)
- Parallel запуск независимых задач (экономит время, не токены)  
- Передавать агентам конкретные файлы, а не "найди сам"
- Max 2-3 агента параллельно

## Ограничения (что НЕ автоматизируется)

- Регистрация/логин на huntr — только вручную
- Финальный сабмит репорта — только вручную  
- Решение о severity — только человек
- Банковские/платёжные данные — никогда
