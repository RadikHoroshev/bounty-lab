# Стандарт составления bug bounty отчёта

> **Для нового агента:** если предыдущий агент прервался — читай этот файл и
> `SUBMISSION_STATUS.md` в этой же директории. Там текущее состояние всех отчётов.
> Продолжай с первого пункта в статусе "READY" без вопросов.

## Принципы

- Отчёт — инженерный документ. Никаких вводных фраз, эпитетов, догадок.
- Каждое утверждение подкреплено кодом, строкой файла или выводом скрипта.
- Все утверждения о воздействии сформулированы как факты, а не как возможные последствия.
- Верификационный скрипт — обязательный артефакт наравне с отчётом.

## Структура отчёта (шаблон)

```
# <ТИП УЯЗВИМОСТИ> via <ВЕКТОР> in <target/repo>

**Target:** Owner/repo
**Version:** ≤ X.Y.Z
**CVSS:** N.N <Severity> (AV:_/AC:_/PR:_/UI:_/S:_/C:_/I:_/A:_)
**CWE:** CWE-NNN: <Name>
**Date:** YYYY-MM-DD

---

## Summary
[1 параграф: что уязвимо, каким образом, что может сделать атакующий]

---

## Root Cause
**File:** path/to/file.py (lines N–M)
[Фрагмент кода с комментарием — только уязвимые строки]

---

## Proof of Concept
### Setup
[Минимальная команда для запуска тестовой среды]

### Exploit
[Точная команда или скрипт, подтверждающий уязвимость]

### Expected Result
[Конкретный вывод или поведение, подтверждающее уязвимость]

---

## Impact
| Сценарий | Воздействие |
|----------|-------------|
| ...      | ...         |

---

## Vulnerable Code Locations
| File | Function | Line | Issue |
|------|----------|------|-------|
| ...  | ...      | ...  | ...   |

---

## Fix
[Минимально необходимое изменение кода с примером]

---

## CVSS Score Justification
**CVSS:3.1/AV:_/AC:_/PR:_/UI:_/S:_/C:_/I:_/A:_ = N.N**
- **AV:** обоснование
- **AC:** обоснование
- ...
```

## Стандарт верификационного скрипта

Файл: `verify_<target>_<type>.py`

### Требования к скрипту

1. **Docstring**: требования к окружению, условия запуска, коды возврата
2. **Зависимости**: только stdlib + минимум внешних пакетов (с обработкой ImportError)
3. **Три шага**:
   - `[1/3]` Sanity check — подтверждение достижимости цели
   - `[2/3]` Exploit — подтверждение уязвимости (RED = уязвим)
   - `[3/3]` Fix — подтверждение что предложенный фикс блокирует (GREEN = исправлен)
4. **Exit codes**:
   - `0` — уязвимость подтверждена
   - `1` — ошибка окружения (зависимость не установлена, цель недоступна)
   - `2` — не воспроизведено (возможно уже исправлено)
5. **Никаких побочных эффектов**: не изменяет файлы, не делает реальных запросов кроме localhost
6. **Цвета**: RED = уязвимо, GREEN = исправлено/нормально, YELLOW = неизвестно

### Проверочный чеклист скрипта

```
[ ] python3 -m py_compile verify_*.py  — синтаксис без ошибок
[ ] Нет неиспользуемых import
[ ] Все внешние зависимости обрабатывают ImportError
[ ] Exit codes задокументированы И реализованы корректно
[ ] Скрипт запущен — RCE/SSRF подтверждён
[ ] Step 3 подтверждает что фикс работает
[ ] Нет жёстко прописанных путей кроме /tmp или localhost
```

## Процесс: от находки до отправки

### Фаза 1 — Черновик отчёта `[SONNET]`

```
1. Открыть шаблон отчёта
2. Заполнить каждый раздел — только факты, строки кода, конкретные действия
3. Проверить:
   - Все file:line ссылки ведут на правильные строки (curl raw.githubusercontent.com)
   - CVSS вектор соответствует реальному воздействию
   - PoC команды скопированы из фактического терминала, не сочинены
4. Сохранить: findings/<target>-<type>.md
```

### Фаза 2 — Верификационный скрипт `[SONNET]`

```
1. Написать скрипт по стандарту (три шага, exit codes, docstring)
2. Сохранить: findings/verify_<target>_<type>.py
3. Записать в SUBMISSION_STATUS.md → статус: "READY FOR QA"
```

### Фаза 3 — QA скрипта `[HAIKU]`

```
Промпт агента: см. AGENT_ROLES.md → "Промпт для Haiku: QA скрипта"

[ ] python3 -m py_compile — синтаксис
[ ] Все import используются
[ ] Exit codes соответствуют docstring
[ ] python3 verify_*.py — Step 2 RED, Step 3 GREEN, exit 0
[ ] Нет неиспользуемых import

Результат → SUBMISSION_STATUS.md: QA PASS / QA FAIL + детали
```

### Фаза 4 — Финальная проверка отчёта `[HAIKU]`

```
Промпт агента: см. AGENT_ROLES.md → "Промпт для Haiku: Финальная проверка отчёта"

[ ] Заголовок точно описывает уязвимость (тип + вектор + репозиторий)
[ ] Target: Owner/repo совпадает с GitHub URL
[ ] Version: проверена в репозитории
[ ] CVSS: все 8 компонент обоснованы в Justification
[ ] Root Cause: file + line — проверены curl raw.githubusercontent.com
[ ] PoC: команды синтаксически валидны
[ ] Fix: конкретный код, не общие слова
[ ] Нет слов: "возможно", "мог бы", "вероятно", "потенциально"

Результат → SUBMISSION_STATUS.md: PREFLIGHT PASS / FAIL + детали
```

### Фаза 5 — Заполнение формы huntr `[HAIKU]`

```
Промпт агента: см. AGENT_ROLES.md → "Промпт для Haiku: Заполнение формы huntr"

Поля (порядок заполнения, источник в отчёте):
1. Repository   ← строка "**Target:**"
2. Version      ← строка "**Version:**"
3. Title        ← первая строка без "#"
4. CVSS Score   ← число из "**CVSS:**"
5. CWE          ← только цифры из "**CWE:**"
6. Description  ← секции Summary + Root Cause
7. Impact       ← секция Impact
8. PoC          ← секция Proof of Concept
9. Attachments  ← verify_*.py

После отправки → записать URL в SUBMISSION_STATUS.md
```

### Фаза 6 — После отправки `[HAIKU]`

```
Промпт агента: см. AGENT_ROLES.md → "Промпт для Haiku: Git commit + push"

1. git add findings/*.md findings/verify_*.py findings/SUBMISSION_STATUS.md
2. git commit + git push
3. Обновить SUBMISSION_STATUS.md: статус → SUBMITTED + huntr URL + дата
```

### Фаза 6b — Обновление memory `[SONNET]`

```
Только Sonnet: обновить memory/project_security_research.md
- Добавить huntr URL
- Обновить статус и bounty potential
```

## Чеклист для передачи между агентами

Если агент прерывается на середине работы, перед остановкой он обязан записать в
`SUBMISSION_STATUS.md` следующее:

```
## ТЕКУЩЕЕ СОСТОЯНИЕ (обновлено: YYYY-MM-DD HH:MM)

### Активная задача
- Что делается прямо сейчас: [заполнить]
- На каком шаге остановились: Фаза N / Шаг N
- Что уже сделано в этой сессии: [список]
- Что осталось: [список]

### Следующее действие для нового агента
[Одна конкретная команда или действие — без вопросов]
```

Новый агент: читай `SUBMISSION_STATUS.md`, выполняй "Следующее действие", продолжай
по стандарту.

## Ошибки которые нельзя допускать

- В PoC написать "возможно, злоумышленник мог бы..." — только конкретные команды
- CVSS без обоснования каждого вектора
- Ссылки на строки кода без проверки через `curl raw.githubusercontent.com/...`
- Верификационный скрипт без step 3 (проверки что фикс работает)
- Отправка отчёта без запуска верификационного скрипта
- Exit code 0 и при уязвимости, и при "не воспроизведено" (различать кодами 0/2)
- Неиспользуемые import в скрипте
